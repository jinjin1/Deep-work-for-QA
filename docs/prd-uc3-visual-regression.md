# UC3: 시각적 회귀 테스트 — 상세 PRD

> **핵심 가치:** "UI가 의도치 않게 깨졌는지 AI가 자동으로 잡아준다"

---

## 1. 문제 정의 & 기회

### 현재 문제
- **시각적 버그의 늦은 발견:** CSS/레이아웃 변경이 의도치 않은 다른 페이지에 영향을 미치지만, 배포 후에야 발견됨
- **수동 비교의 한계:** 디자이너가 디자인 시안과 구현을 눈으로 비교하는 것은 미세한 차이를 놓치기 쉬움
- **픽셀 비교의 노이즈:** 기존 도구(Percy, Chromatic)는 픽셀 단위 비교로 동적 컨텐츠(시간, 광고 등) 때문에 오탐이 과다
- **CI/CD 의존성:** 대부분의 시각적 회귀 도구가 CI 파이프라인 연동을 요구해 도입 장벽이 높음

### 기회
- **브라우저 확장 기반 접근:** CI/CD 없이 팀원 누구나 수동으로 베이스라인을 설정하고 비교할 수 있어 도입 장벽이 극히 낮음
- **AI 시맨틱 비교:** 픽셀이 아닌 의미 단위(레이아웃, 텍스트, 색상, 요소 유무)로 비교하여 오탐 감소
- **의도/버그 분류:** AI가 변경을 "의도적"과 "회귀(버그)"로 분류하여 리뷰 시간 단축
- UC1과 결합하면 시각적 회귀 발견 → 버그 리포트 생성까지 자동화

## 2. 유저 스토리

### 디자이너 (현우)
```
AS A Designer
I WANT TO 주요 페이지들의 기준 스크린샷을 저장해두고, 배포 후 변경사항을 자동으로 비교
SO THAT 디자인 의도와 다르게 구현된 부분을 즉시 발견하고 수정 요청할 수 있다
```

### QA 엔지니어 (수진)
```
AS A QA Engineer
I WANT TO 배포 전후로 주요 페이지의 시각적 변경을 자동으로 감지하고, AI가 의도적 변경과 버그를 구분
SO THAT 모든 페이지를 수동으로 확인하지 않아도 시각적 회귀를 놓치지 않을 수 있다
```

### 프론트엔드 개발자 (민수)
```
AS A Frontend Developer
I WANT TO CSS/컴포넌트 변경 후 영향받는 페이지의 시각적 차이를 확인
SO THAT 의도치 않은 사이드이펙트를 배포 전에 발견하고 수정할 수 있다
```

## 4. 유저 플로우 (단계별 상세)

### Flow 1: 베이스라인 설정

```
[1] 팀원이 주요 페이지 방문 (ex: 홈페이지)
         │
[2] Deep Work 확장 아이콘 클릭 → "시각적 테스트" 탭 선택
         │
[3] "베이스라인 저장" 클릭
    ├─ 풀페이지 스크린샷 캡처
    ├─ 뷰포트 크기 자동 감지
    └─ 페이지 URL 자동 기록
         │
[4] 베이스라인 정보 입력
    ├─ 이름: "홈페이지 - Desktop" (자동 제안)
    ├─ 무시 영역 설정 (optional): 동적 컨텐츠 영역 드래그
    └─ 추가 뷰포트 (optional): 태블릿/모바일도 캡처
         │
[5] "저장" → 베이스라인 등록 완료
    └─ 프로젝트의 베이스라인 목록에 추가
```

### Flow 2: 시각적 비교 실행

```
[1] 배포 후, 팀원이 베이스라인이 있는 페이지 방문
         │
[2] 확장에서 자동 알림: "이 페이지의 베이스라인이 있습니다. 비교할까요?"
    (또는 수동으로 "시각적 테스트" → "비교 실행" 클릭)
         │
[3] 현재 페이지 풀페이지 스크린샷 캡처
         │
[4] 서버로 업로드 → AI 비교 분석 시작
    ├─ 픽셀 비교 (차이 영역 감지)
    ├─ AI 시맨틱 분석 (변경 유형 분류)
    └─ 의도적/회귀 분류
         │
[5] 비교 결과 화면 표시 (Side Panel 또는 새 탭)
    ├─ 전체 상태: ✅ No Change / 🟡 Changes Detected / 🔴 Regression Found
    ├─ 변경 영역 목록 + 분류
    └─ 베이스라인 | 현재 | Diff 3-way 비교 뷰
         │
[6] 각 변경에 대해 팀원 액션:
    ├─ ✅ "의도적 변경" → 베이스라인 업데이트 제안
    ├─ 🔴 "회귀 맞음" → "버그 리포트 생성" (UC1 연동)
    └─ 🤔 "불확실" → 다른 팀원에게 공유
```

### Flow 3: 일괄 비교

```
[1] 웹 대시보드에서 프로젝트의 "시각적 테스트" 페이지 접근
         │
[2] "전체 비교 실행" 클릭
         │
[3] 서버가 모든 베이스라인 URL을 헤드리스 브라우저로 캡처
    (Puppeteer/Playwright 기반)
         │
[4] 각 베이스라인과 현재 스크린샷 비교 → AI 분석
         │
[5] 비교 결과 대시보드:
    ├─ ✅ 변경 없음: 8건
    ├─ 🟡 의도적 변경: 3건
    ├─ 🔴 회귀 감지: 2건
    └─ 🤔 확인 필요: 1건
         │
[6] 각 항목 클릭 → 상세 비교 화면
```

## 5. AI 시각적 비교 엔진 설계

### 비교 파이프라인

```
입력:
├─ 베이스라인 스크린샷 (PNG, 풀페이지)
├─ 현재 스크린샷 (PNG, 풀페이지)
├─ 무시 영역 좌표 (있는 경우)
└─ 페이지 URL + 뷰포트 정보

    │
    ▼
Stage 1: 픽셀 비교 (즉시)
├─ 이미지 정렬 (크기 차이 시 리사이즈)
├─ 무시 영역 마스킹
├─ 픽셀 단위 diff 계산
├─ 변경 영역 클러스터링 (인접 픽셀 그룹핑)
└─ 각 클러스터의 바운딩 박스 추출

    │
    ▼
Stage 2: AI 시맨틱 분석 (비동기)
├─ Vision AI에 베이스라인 + 현재 + diff 영역 전달
├─ 각 변경 영역의 유형 분류:
│   ├─ layout_shift: 요소 위치 이동
│   ├─ text_change: 텍스트 내용 변경
│   ├─ color_change: 색상/배경 변경
│   ├─ element_missing: 요소 사라짐
│   ├─ element_added: 요소 추가됨
│   ├─ size_change: 요소 크기 변경
│   └─ other: 기타
├─ 각 변경의 의도/회귀 분류 + confidence score
└─ 각 변경에 대한 자연어 설명 생성

    │
    ▼
출력:
├─ diff 이미지 (변경 영역 하이라이트)
├─ VisualChange[] (영역, 유형, 분류, 설명)
└─ overall_status (no_change / intentional / regression / mixed)
```

### Vision AI 프롬프트 구성

```
System: "당신은 UI/UX 전문가입니다. 웹 페이지의 두 스크린샷을 비교하여
         각 변경 사항을 분석하세요."

Input:
  - Image 1: 베이스라인 스크린샷
  - Image 2: 현재 스크린샷
  - Diff regions: [{ x, y, width, height }] (픽셀 비교로 감지된 영역)

Task:
  각 diff region에 대해:
  1. 변경 유형을 분류하세요 (layout_shift, text_change, color_change,
     element_missing, element_added, size_change, other)
  2. 이 변경이 의도적인지(intentional) 회귀(regression)인지 판단하세요
     - 의도적: 기능 개선, 디자인 업데이트, 컨텐츠 변경 등
     - 회귀: 레이아웃 깨짐, 요소 겹침, 텍스트 잘림, 정렬 오류 등
  3. 판단 근거와 함께 자연어 설명을 작성하세요

Output Format: JSON
  {
    "changes": [
      {
        "region_index": 0,
        "type": "layout_shift",
        "classification": "regression",
        "confidence": 0.85,
        "description": "네비게이션 메뉴가 오른쪽으로 15px 이동하여 마지막 메뉴 항목이 화면 밖으로 잘렸습니다."
      }
    ],
    "overall_assessment": "regression",
    "summary": "네비게이션 레이아웃 회귀가 감지되었습니다. CSS 변경으로 인한 사이드이펙트로 보입니다."
  }
```

### 분류 기준 가이드

| 의도적 변경 (Intentional) | 회귀/버그 (Regression) |
|--------------------------|----------------------|
| 새 기능/버튼 추가 | 요소가 겹침 (overlap) |
| 디자인 색상/폰트 업데이트 | 텍스트가 잘림 (overflow) |
| 컨텐츠 텍스트 변경 | 레이아웃 깨짐 (broken layout) |
| 의도적 레이아웃 재배치 | 요소가 의도치 않게 사라짐 |
| 이미지/아이콘 업데이트 | 정렬 오류 (misalignment) |
| 반응형 개선 | 여백/패딩 비정상 (spacing broken) |

## 6. 데이터 모델 (UC3 고유)

> 공통 모델(Baseline, VisualDiff, VisualChange)은 `prd-overview.md` 참조. 여기서는 UC3 고유 모델만 기술.

### IgnoreRegion (무시 영역)
```
IgnoreRegion {
  id: UUID
  baseline_id: UUID (FK → Baseline)
  region: { x, y, width, height }  // 비교에서 제외할 영역
  reason: string?                   // "시계 위젯", "광고 영역" 등
}
```

### ComparisonRun (일괄 비교 실행)
```
ComparisonRun {
  id: UUID
  project_id: UUID (FK → Project)

  // 실행 메타
  trigger: enum (manual, scheduled)   // MVP: manual만
  status: enum (pending, running, completed, failed)

  // 결과 요약
  total_baselines: number
  no_change_count: number
  intentional_count: number
  regression_count: number
  uncertain_count: number

  // 연결
  visual_diff_ids: UUID[]            // 이 실행에서 생성된 VisualDiff 목록

  created_by: UUID (FK → User)
  created_at: timestamp
  completed_at: timestamp?
}
```

## 7. API 엔드포인트

### 베이스라인 CRUD

```
POST   /v1/baselines                       # 베이스라인 생성
GET    /v1/baselines                        # 베이스라인 목록 (프로젝트별)
GET    /v1/baselines/:id                    # 베이스라인 상세
PUT    /v1/baselines/:id                    # 베이스라인 업데이트 (스크린샷 교체)
DELETE /v1/baselines/:id                    # 베이스라인 삭제
```

### 무시 영역

```
POST   /v1/baselines/:id/ignore-regions     # 무시 영역 추가
GET    /v1/baselines/:id/ignore-regions      # 무시 영역 목록
DELETE /v1/baselines/:id/ignore-regions/:rid # 무시 영역 삭제
```

### 시각적 비교

```
POST   /v1/visual-diffs                     # 단일 비교 실행
GET    /v1/visual-diffs/:id                 # 비교 결과 상세
POST   /v1/visual-diffs/:id/approve         # 변경 승인 (베이스라인 업데이트)
POST   /v1/visual-diffs/:id/bug-report      # 회귀에서 버그 리포트 생성 (UC1 연동)
```

### 일괄 비교

```
POST   /v1/comparison-runs                  # 일괄 비교 실행
GET    /v1/comparison-runs                  # 비교 실행 히스토리
GET    /v1/comparison-runs/:id              # 비교 실행 상세 (결과 포함)
```

## 9. 성공 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 프로젝트당 베이스라인 수 | > 10개 | 활성 프로젝트의 평균 베이스라인 수 |
| 주간 비교 실행 횟수 | > 3회/프로젝트 | 프로젝트당 주간 비교 실행 횟수 |
| 변경 감지율 | > 90% | 실제 시각적 변경 중 감지된 비율 |
| 의도/회귀 분류 정확도 | > 75% | AI 분류가 유저 판단과 일치한 비율 |
| 오탐률 (false positive) | < 20% | 변경으로 감지되었으나 실제 변경 아닌 비율 |
| 비교 → 버그 리포트 전환율 | > 20% | regression 분류 중 버그 리포트로 생성된 비율 |
| 비교 분석 처리 시간 | < 30초 | 스크린샷 업로드 → 결과 표시까지 시간 |

|--------|------|------|
| 동적 컨텐츠로 인한 오탐 과다 | 유저 신뢰도 하락 | 무시 영역 설정, AI 시맨틱 비교로 동적 컨텐츠 자동 필터링 |

| 반응형 페이지에서 뷰포트 차이로 오탐 | 비교 부정확 | 비교 시 동일 뷰포트 강제, 뷰포트 정보를 베이스라인에 저장 |

## 12. 기술 구현 참고

### 스크린샷 캡처 방식
```
브라우저 확장 (수동 비교):
├─ chrome.tabs.captureVisibleTab() → 현재 보이는 영역
├─ 풀페이지: 스크롤하며 여러 캡처 → 서버에서 스티칭
└─ 뷰포트 크기는 window.innerWidth/innerHeight

일괄 비교 (서버 기반):
├─ Puppeteer/Playwright로 헤드리스 브라우저 실행
├─ 베이스라인 URL 접속 → page.screenshot({ fullPage: true })
├─ 뷰포트 크기는 베이스라인 설정값 사용
└─ 인증이 필요한 페이지: 유저가 제공한 쿠키/토큰 사용
```

### 픽셀 비교 알고리즘
```
1. 이미지 정렬
   ├─ 두 이미지 크기 맞춤 (작은 쪽에 padding)
   └─ 무시 영역 마스킹 (투명 처리)

2. 픽셀 diff
   ├─ 각 픽셀의 RGB 차이 계산
   ├─ 임계값(threshold) 이상 차이 → 변경 픽셀로 마킹
   └─ 임계값: 기본 10% (설정 가능)

3. 영역 클러스터링
   ├─ 변경 픽셀을 인접 그룹으로 클러스터링
   ├─ 너무 작은 클러스터(< 100px²) 필터링
   └─ 각 클러스터의 바운딩 박스 추출

4. diff 이미지 생성
   ├─ 베이스라인 위에 변경 영역을 반투명 컬러로 하이라이트
   └─ 바운딩 박스 테두리 표시

라이브러리: pixelmatch (Node.js) 또는 동등 기능
```
